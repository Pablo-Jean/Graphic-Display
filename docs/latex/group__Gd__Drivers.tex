\doxysubsection{Detailed Description}
Driver definition for the Graphic Display library, this driver is the main reason why we can use any display with this library, you just need to create functions that are compatible with this structure. 

\hypertarget{group__Gd__Drivers_autotoc_md0}{}\doxysubsubsection{$>$$>$ Step by Step\+:}\label{group__Gd__Drivers_autotoc_md0}
\hypertarget{group__Gd__Drivers_autotoc_md1}{}\doxyparagraph{1. Create a folder inside drivers with the name of driver device.}\label{group__Gd__Drivers_autotoc_md1}
Create a folder to place the source code of your driver, in this way, we can keep a clean and nice repository.\hypertarget{group__Gd__Drivers_autotoc_md2}{}\doxyparagraph{2. Create the .\+h and .\+c file for you device.}\label{group__Gd__Drivers_autotoc_md2}
Suppose we create a driver for a fictional ss9090 device, we will create {\bfseries{ss9090.\+h}} and {\bfseries{ss9090.\+c files}}.\hypertarget{group__Gd__Drivers_autotoc_md3}{}\doxyparagraph{3. In header file, create the handle structure}\label{group__Gd__Drivers_autotoc_md3}
The handle structure is used to keep data of a display, with this approach, we can have many displays our project requires, and this displays can be the same or different.


\begin{DoxyCode}{0}

The most common parameters are the\+:
\begin{DoxyItemize}
\item Height and Width of the display;
\item Frame Buffer pointer;
\item Communication Functions (For SPI, we have the Chip Select and SPI transfer, For I2C, commonly has only the write function).
\end{DoxyItemize}

Other parameters, that\textquotesingle{}s are optional, but recommended, if a function to mutex lock and unlock (useful for RTOS environments that need threadsafe), and Delay Milliseconds routine, some times, delay is mandatory in initialization.

Remember, the driver must be not glued with any chipset, this library must be compatible with any microcontroller.\hypertarget{group__Gd__Drivers_autotoc_md4}{}\doxyparagraph{4. Create routines}\label{group__Gd__Drivers_autotoc_md4}
